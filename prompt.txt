# Project: Personal CMS for Hugo + Turso

## Overview

I have an existing personal website built with Hugo.

I want to build a completely separate repository that acts as a private CMS (admin dashboard) for my website.

This CMS will become the **single source of truth** for structured content.

The Hugo website will remain completely static and will consume data directly from Turso using Hugo Content Adapters during the build.

No generated Markdown should be committed to Git.

No generated JSON should be written to disk.

The Hugo website is read-only.

The CMS is the only application that writes to the database.

---

# Overall Architecture

```text
                    Turso (libSQL)
                           │
          ┌────────────────┴────────────────┐
          │                                 │
          ▼                                 ▼
     Personal CMS                    Hugo Website
     (Next.js)                  (Content Adapters)
          │                                 │
          └────────────────┬────────────────┘
                           ▼
                     Static Website
```

The Hugo website should never contain business logic for editing content.

The CMS should never render the public website.

Each application has a single responsibility.

# Coding Standards

* Strict TypeScript
* Reusable components
* Good naming
* Clean architecture
* Modular code
* No unnecessary abstractions
* No overengineering
* Follow current Next.js best practices

The resulting CMS should feel like a professional internal tool built specifically for managing a personal website, with a strong foundation for future expansion.

This is a personal CMS, not a product. Optimize for a single trusted user, not for multiple users or SaaS use cases. Favor simplicity, maintainability, and fast workflows over configurability. Avoid features that only make sense in multi-user systems.

Design Philosophy: This application is used daily by a single power user. Prioritize speed, clarity, keyboard navigation, and information density over visual decoration. The interface should feel html friendly, utalitarian. Take inspiration from hackernews design, but instead of having only one color orange add thenming options. Use icons wherever possible. Make the website resposive to mobile, tabs and desktop. Avoid enterprise dashboard aesthetics, oversized cards, unnecessary whitespace, loud gradients, and decorative animations. Every screen should help the user accomplish a task with as few clicks as possible.
