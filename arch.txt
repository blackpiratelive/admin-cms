# Architecture Evolution: Transform the Personal CMS into a Personal Knowledge Platform

## Objective

This document describes the next architectural evolution of the Personal CMS.

The CMS is no longer just a content management system for a Hugo website.

It should evolve into a **Personal Knowledge Platform** that manages, connects, enriches and publishes every aspect of my digital life while maintaining strict ownership boundaries between provider-owned data and personal metadata.

The CMS should remain modular, scalable, provider-based, and entity-driven.

The current architecture (Microblogs, Gallery, Sync Center, Libraries, Hugo Content Adapters, Turso, Cloudflare R2, Trakt, Last.fm, etc.) should remain unchanged unless specifically mentioned below.

Everything in this document extends the existing architecture.
Make sure no data loss from the database occurs. add migration scripts wherever necessary.

---

# Development Principles

Always prefer:

Reusable abstractions

Entity-driven architecture

Provider separation

Strong typing

Composition over duplication

Event-driven workflows

Background jobs for expensive operations

Keyboard-first UX

Fast search

Dark mode first

Minimal UI

Scalable database design

Avoid feature-specific hacks.

Every new feature should integrate with the existing architecture.

# Long-Term Goal

The finished application should not feel like a CMS.

It should feel like a **personal operating system** and **knowledge graph**.

It should know:

* What I created
* What I watched
* What I listened to
* What I photographed
* Where I traveled
* What I built
* What I learned
* How everything relates to everything else

The Hugo website is simply one way of presenting this information.

The CMS is the canonical source of personal knowledge, relationships, metadata and workflows, built on a clean, modular, extensible architecture that can evolve for many years without major redesigns.
